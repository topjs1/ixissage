# RISKS_AND_LIMITATIONS.md

## 프로젝트 성격의 한계

`ixissage`는 상용 보안 제품이 아니다.

이 프로젝트는 인공지능 과목 수업을 위한 교육용 데모다. 실제 사용자에게 완전한 스미싱 방어 기능을 제공하는 것이 목표가 아니다. Google Play Store 배포도 목표가 아니다.

따라서 발표나 README에서 다음을 명확히 해야 한다.

- 교육용 AI 데모 앱
- 실제 보안 제품 아님
- 스미싱 탐지를 보장하지 않음
- 모델 오분류 가능
- SMS 권한 기능은 선택 사항

## 룰 기반 탐지 금지로 인한 설계 리스크

이 프로젝트는 룰 기반 탐지를 금지한다.

금지 예:

- `"택배"`라는 단어가 있으면 위험
- `"미납"`이라는 단어가 있으면 위험
- URL이 있으면 위험
- `"환급"`이 있으면 위험
- `"본인인증"`이 있으면 위험

이 제약은 단순한 구현 제한이 아니라 프로젝트의 핵심 원칙이다. 목표는 사람이 만든 규칙이 아니라 AI 모델이 데이터에서 학습한 확률로 분류하는 것이다.

리스크:

- 간단한 키워드 규칙보다 구현이 오래 걸릴 수 있다.
- 모델이 특정 유형의 메시지를 잘못 분류할 수 있다.
- 수업 데모에서 왜 rule을 쓰지 않았는지 설명이 필요할 수 있다.

대응:

- 프로젝트 문서와 발표에서 supervised text classification 문제라고 설명한다.
- 평가 지표와 오류 분석으로 모델의 장단점을 보여준다.
- 확률 기반 배지를 사용해 모델 불확실성을 표현한다.

## 데이터셋 리스크

`meal-bbang/Korean_message`는 프로젝트에 적합하지만 몇 가지 주의점이 있다.

확인된 이슈:

- Hugging Face 카드에는 `label 1`과 `label 2` 의미만 설명되어 있다.
- 실제 CSV에는 `label 3`도 존재한다.
- Hugging Face viewer에는 train split만 표시된다.
- 명확한 라이선스가 표시되어 있지 않다.
- 일부 데이터는 생성형 AI 또는 여러 소스에서 만들어졌을 수 있다.

대응:

- EDA에서 `label 3` 샘플을 별도로 분석한다.
- 첫 baseline은 `label 1`과 `label 2`만 사용해도 된다.
- `label 3`을 포함하는 실험은 처리 방침을 문서화한 뒤 진행한다.
- train/dev/test split을 직접 만든다.
- 상용 사용이 아니라 수업 프로젝트임을 명확히 한다.

## 라벨 품질 리스크

스미싱 데이터셋은 라벨 오류가 있을 수 있다.

예상 문제:

- 정상 알림 문자가 스미싱처럼 보일 수 있음
- 스미싱 문자가 정상 안내처럼 보일 수 있음
- 생성형 AI로 만든 문장이 실제 SMS 분포와 다를 수 있음
- 같은 메시지가 중복되어 있을 수 있음

대응:

- 중복 제거 여부 확인
- 라벨별 샘플 수동 검토
- false positive와 false negative 예시 분석
- 결과 리포트에 데이터 한계 명시

## 모델 성능 리스크

모델은 항상 틀릴 수 있다.

False Positive:

- 정상 문자를 스미싱으로 잘못 판단
- 사용자가 앱을 불신할 수 있음

False Negative:

- 스미싱 문자를 정상으로 잘못 판단
- 실제 위험을 놓칠 수 있음

대응:

- accuracy만 보지 않는다.
- precision, recall, F1-score를 함께 본다.
- confusion matrix를 출력한다.
- threshold별 성능 변화를 비교한다.
- 모델 결과를 절대적 판정이 아니라 확률 기반 참고 정보로 표시한다.

## Android 권한 리스크

실제 SMS를 읽으려면 `READ_SMS` 권한이 필요하다.

리스크:

- Android 버전에 따라 동작이 다를 수 있다.
- 권한 요청이 사용자에게 부담을 줄 수 있다.
- Play Store 정책상 SMS 권한은 매우 제한적이다.
- 권한이 거부되면 실제 SMS 기능이 동작하지 않는다.

대응:

- 초기 버전은 `sample_messages.json`만 사용한다.
- 실제 SMS 기능은 후반부 선택 기능으로 둔다.
- 권한이 막히면 sample dataset mode로 대체한다.
- 프로젝트 완성 기준에 실제 SMS 기능을 필수로 넣지 않는다.

## 개인정보 리스크

SMS 본문은 매우 민감한 개인정보다.

금지:

- 문자 본문을 외부 서버로 전송
- 문자 본문을 외부 AI API에 전송
- 문자 본문을 Logcat에 출력
- 실제 SMS 데이터를 Git에 커밋
- 사용자 SMS를 분석 도구로 전송

대응:

- 샘플 데이터 중심으로 개발한다.
- 실제 SMS 기능을 넣더라도 로컬에서만 처리한다.
- 로그에는 메시지 ID나 길이 정도만 남긴다.
- 발표 자료에는 개인정보 보호 원칙을 명시한다.

## TFLite/LiteRT 변환 리스크

Transformer 모델을 Android에서 직접 실행하는 것은 어려울 수 있다.

리스크:

- 모델 크기가 클 수 있다.
- tokenizer를 Android에서 구현해야 할 수 있다.
- 변환이 실패할 수 있다.
- 추론 속도가 느릴 수 있다.
- 앱 용량이 커질 수 있다.

대응:

- 먼저 Python inference 결과를 JSON으로 만들어 Android에 표시한다.
- 온디바이스 추론은 선택 기능으로 둔다.
- 시간이 부족하면 변환 검토 결과만 문서화한다.

## 데모 해석 리스크

사용자가 앱 결과를 실제 보안 판정으로 오해할 수 있다.

대응:

- 앱과 발표에서 교육용 데모임을 표시한다.
- `스팸 경고` 배지는 모델 확률 기반 표시임을 설명한다.
- 모델이 틀릴 수 있음을 리포트에 명시한다.
- 실제 금융/개인정보 판단은 공식 기관 확인이 필요하다고 안내한다.

## 프로젝트 완료 기준의 한계

이 프로젝트는 다음을 완료하면 성공으로 본다.

- 데이터셋 조사
- EDA
- baseline 모델
- Transformer 모델
- 평가 지표
- 오류 분석
- inference.py
- sample_messages.json
- Android 데모 앱

다음은 선택 기능이다.

- 실제 SMS 읽기
- TFLite/LiteRT 변환
- 온디바이스 실시간 추론

핵심은 AI 모델이 문자 메시지를 분류하고, Android 앱이 그 확률을 보여주는 것이다. 룰 기반 탐지 기능을 추가해 성능을 꾸미는 것은 이 프로젝트의 목표와 맞지 않는다.
